## Overview

MuscoviteHarbor is a domain-driven application generated by Muscovite. This guide provides operational instructions for AI agents working on the codebase.

## Quick Start

1. Install dependencies: `conan install --profile conan_profiles/debug .`
2. Configure: `cmake --preset debug`
3. Build: `cmake --build --preset debug`
4. Test: `ctest --preset unit --output-on-failure`

## Build Commands

```bash
conan install --profile conan_profiles/debug .
cmake --preset debug
cmake --build --preset debug
```

Presets: `debug`, `release`, `asan`, `tsan`, `coverage`

## Test Commands

```bash
ctest --preset unit --output-on-failure   # Unit tests
ctest --preset debug --output-on-failure   # All tests
```

All test gates are **blocking** — never `warn`, always `fail`.

## Database

- PostgreSQL with muscdb migrations
- Use `TEXT` (no `VARCHAR`), `TIMESTAMPTZ` (no `TIMESTAMP`)
- Every table has `id` UUID primary key
- No stored procedures — domain logic in application layer

```bash
cmake --build --preset debug --target docker-start   # Start PostgreSQL
cmake --build --preset debug --target db-setup-*     # Deploy migrations
cmake --build --preset debug --target db-fresh-*     # Drop + redeploy
```

## gRPC

- All RPCs are asynchronous — no sync stubs
- Deadline propagation on every call
- Proto files are generated — do not edit directly
- Error mapping uses standard gRPC status codes

## Qt Frontend

### Qt Widgets

- Model/View separation for all data display
- All strings translatable via `tr()`
- No synchronous gRPC from UI thread

### QML

- Versioned QML module imports
- ViewModel pattern for data binding
- Async gRPC via signal/slot integration

## Sidecar Protocol

Sidecar files are generated once, then user-owned.

- `MUSCOVITE:SIDECAR:UNMODIFIED` sentinel — remove to claim ownership
- Interface files in `generated/plugins/` — regenerated every run
- Sidecar files in `sidecars/plugins/` — user-owned after sentinel removal

## Workflow

- Feature branches: `feature/ftr-NNN-short-description`
- No direct commits to `main`
- Run `tools/lint-ddd.sh` before committing `.ddd`/`.mus` changes
- Document changes alongside code
